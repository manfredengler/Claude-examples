Cada concepto del ecosistema Claude Code explicado con ejemplos reales y ejecutables. Empezamos con las primitivas básicas y terminamos combinándolas todas.

01

Tools

Primitiva

Los tools son las acciones concretas que Claude puede ejecutar: leer un archivo, correr bash, editar código, buscar en la web. Sin tools, Claude solo puede devolver texto. Con tools, puede actuar sobre el sistema.

💡
Analogía: Si Claude es un programador, los tools son su teclado, terminal y editor. Los built-in tools (Read, Bash, Edit, Write) siempre están disponibles. Tools adicionales vienen por MCP o por el Agent SDK.
Claude llama al tool 
{
  "tool": "Bash",
  "input": {
    "command": "git log --oneline -5",
    "description": "Last 5 commits"
  }
}
El sistema responde 
a3f1c2e fix: null pointer in auth
9b2e451 feat: add retry logic
c77f830 docs: update README
4d1a9f2 refactor: split services
8eae001 init: project scaffold
Ejemplo real — tool Edit src/config.ts 
// Claude recibe: "cambia el timeout a 30 segundos"
// Claude llama a Edit con old_string / new_string:

{
  "tool": "Edit",
  "input": {
    "file_path": "/src/config.ts",
    "old_string": "const TIMEOUT_MS = 5000;",
    "new_string": "const TIMEOUT_MS = 30_000;"
  }
}
Tools built-in más usados 
Read      → leer archivos (con offset/limit para archivos grandes)
Write     → crear archivos nuevos o reescritura completa
Edit      → reemplazar string exacto dentro de un archivo
Bash      → ejecutar comandos shell (git, npm, curl…)
Agent     → lanzar un sub-agente con su propio contexto
WebSearch → buscar en internet (requiere permiso)
WebFetch  → leer una URL concreta
02

Commands

Built-in + Custom

Los commands son shortcuts prefijados con / que se pueden escribir en el chat. Los built-in vienen con Claude Code. Los custom se crean como archivos .md en .claude/commands/.

Comandos built-in principales 
/help          # lista de comandos disponibles
/clear         # limpia el contexto de la conversación
/compact       # compacta el historial para ahorrar tokens
/memory        # edita el archivo CLAUDE.md del proyecto
/config        # abre la configuración interactiva
/doctor        # diagnóstico del entorno Claude Code
/cost          # muestra el costo de tokens de la sesión
/review        # code review del diff actual (skill)
/init          # genera CLAUDE.md inicial para el repo
Crear un comando custom .claude/commands/fix-ts.md 
---
description: "Corrige todos los errores de TypeScript en el proyecto"
---

Corre `npx tsc --noEmit` y captura la salida.
Por cada error encontrado:
1. Lee el archivo afectado con el tool Read
2. Entiende el contexto del error
3. Aplica la corrección mínima con Edit
4. Vuelve a correr tsc para verificar que se resolvió

Al final lista cuántos errores había y cuántos se corrigieron.
⚡
Una vez creado el archivo, el comando /fix-ts queda disponible en cualquier sesión dentro de ese proyecto. Puede recibir argumentos: /fix-ts src/auth.ts.
Comando con argumento $ARGUMENTS .claude/commands/explain.md 
---
description: "Explica una función o archivo en detalle"
---

Lee el archivo o función indicada: $ARGUMENTS

Explica:
- Qué hace (propósito de alto nivel)
- Cómo funciona (flujo paso a paso)
- Por qué está escrito así (decisiones de diseño no obvias)
- Posibles puntos de mejora

Sé conciso. Usa ejemplos cuando ayuden.
03

Hooks

Automatización

Los hooks son comandos shell que el harness ejecuta automáticamente en respuesta a eventos de Claude: antes de usar un tool, después de usarlo, al iniciar sesión, etc. Claude no los controla — los ejecuta el sistema.

Eventos disponibles 
PreToolUse    → antes de que Claude ejecute cualquier tool
PostToolUse   → después de que un tool termina
SessionStart  → al inicio de cada sesión
Stop          → cuando Claude termina su turno
Hook: lint automático tras cada Edit .claude/settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix \"${TOOL_INPUT_FILE_PATH}\" 2>&1 | head -20"
          }
        ]
      }
    ]
  }
}
Hook: log de todos los comandos Bash .claude/settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[$(date -u +%H:%M:%S)] BASH: $TOOL_INPUT_COMMAND\" >> .claude/bash-audit.log"
          }
        ]
      }
    ]
  }
}
Hook SessionStart: setup del entorno .claude/settings.json 
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm install --silent && echo 'deps ready'"
          }
        ]
      }
    ]
  }
}
🔒
Seguridad: los hooks pueden bloquear un tool devolviendo exit code distinto de 0. Útil para prevenir rm -rf o push a main sin confirmación.
04

Skills

Web / Avanzado

Las skills son comandos slash con inteligencia integrada. A diferencia de los commands simples (texto plano), las skills tienen frontmatter que las registra en el sistema, pueden incluir lógica condicional y se usan especialmente en Claude Code on the Web.

Skill: security review .claude/skills/security-review.md 
---
description: >
  Revisa el diff actual buscando vulnerabilidades de seguridad
  (OWASP Top 10, secrets expuestos, inyecciones, etc.)
trigger: manual
---

## Pasos

1. Corre `git diff HEAD` para ver los cambios actuales
2. Para cada archivo modificado:
   - Lee el archivo completo con Read
   - Analiza el diff en contexto
3. Busca específicamente:
   - SQL/command injection
   - Secrets o credenciales hardcodeadas
   - XSS en templates HTML
   - Dependencias con CVEs conocidos (`npm audit`)
   - Permisos excesivos en archivos
4. Por cada hallazgo: indica archivo, línea, severidad y fix sugerido
5. Termina con un resumen: PASS / WARN / FAIL
Skill: deploy con validaciones .claude/skills/deploy.md 
---
description: "Deploy a producción con checklist automático"
trigger: manual
---

Antes de deployar, valida que:
- [ ] Tests pasan: `npm test`
- [ ] Build compila: `npm run build`
- [ ] No hay secrets en el código: `git log -p | grep -i "api_key\|password\|secret"`
- [ ] La rama es `main` o `release/*`

Si todo pasa, ejecuta el deploy: `npm run deploy`
Si algún check falla, detente y reporta qué falló.
🎯
Skills vs Commands: Los commands son instrucciones. Las skills son procedimientos con contexto, validaciones y flujo de decisión. Se invocan igual (/deploy), pero tienen más poder expresivo.
05

MCP Servers

Extensibilidad

MCP (Model Context Protocol) permite conectar Claude a sistemas externos como GitHub, bases de datos, Slack, APIs internas. Cada servidor MCP expone tools adicionales que Claude puede llamar igual que los built-in.

Configurar MCP en el proyecto .claude/settings.json 
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "${DATABASE_URL}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    }
  }
}
Claude llama tool MCP GitHub 
{
  "tool": "mcp__github__create_pull_request",
  "input": {
    "owner": "acme",
    "repo": "api",
    "title": "fix: auth timeout",
    "head": "feature/fix-auth",
    "base": "main",
    "body": "Increases timeout to 30s"
  }
}
Respuesta del servidor MCP 
{
  "number": 142,
  "url": "https://github.com/acme/api/pull/142",
  "state": "open",
  "created_at": "2026-06-02T10:30:00Z"
}
Crear un MCP server propio (Node.js) mcp-server/index.ts 
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "mi-empresa", version: "1.0.0" });

// Tool personalizado: consultar métricas internas
server.tool(
  "get_metrics",
  "Obtiene métricas de rendimiento del sistema",
  {
    service: z.string().describe("nombre del servicio"),
    range: z.enum(["1h", "24h", "7d"]).default("1h"),
  },
  async ({ service, range }) => {
    const data = await fetchInternalMetrics(service, range);
    return {
      content: [{ type: "text", text: JSON.stringify(data, null, 2) }],
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
06

Agents

Paralelismo

Un agent es una instancia de Claude que corre de forma independiente, con su propio contexto y set de tools. El agente padre los lanza con el tool Agent y recibe su resultado como string cuando terminan.

🧠
Cuándo usar agents: tareas largas que consumirían todo el contexto, trabajo paralelo independiente, o cuando quieres aislar una búsqueda para que no contamine el contexto principal.
Lanzar 3 agentes en paralelo 
// Esto ocurre internamente cuando Claude decide paralelizar:
// Los 3 Agent calls van en el mismo mensaje → corren a la vez

Agent({
  description: "Analiza el módulo de auth",
  prompt: `Lee todos los archivos en src/auth/.
           Lista: funciones exportadas, dependencias externas,
           posibles vulnerabilidades. Responde en <200 palabras.`
});

Agent({
  description: "Analiza el módulo de pagos",
  prompt: `Lee todos los archivos en src/payments/.
           Lista: funciones exportadas, dependencias externas,
           posibles vulnerabilidades. Responde en <200 palabras.`
});

Agent({
  description: "Analiza el módulo de usuarios",
  prompt: `Lee todos los archivos en src/users/.
           Lista: funciones exportadas, dependencias externas,
           posibles vulnerabilidades. Responde en <200 palabras.`
});
Agente con worktree aislado 
// isolation: "worktree" crea una copia git aislada
// El agente puede modificar archivos sin afectar la rama actual

Agent({
  description: "Migrar tests a Vitest",
  isolation: "worktree",
  prompt: `Migra todos los archivos *.test.ts de Jest a Vitest:
  1. Reemplaza imports de jest por vitest
  2. Actualiza mocks y spies a la API de Vitest
  3. Corre los tests para verificar que pasan
  4. Haz commit de los cambios
  Rama base: main. Trabaja en todos los archivos de forma exhaustiva.`
});
Agente en background 
// run_in_background: true → no bloquea, se recibe notificación al terminar

Agent({
  description: "Generar documentación completa",
  run_in_background: true,
  prompt: `Genera JSDoc para todas las funciones públicas en src/.
  Usa Read para leer cada archivo, Edit para añadir los comentarios.
  Prioriza claridad sobre exhaustividad.`
});

// El agente principal puede seguir trabajando mientras tanto
07

Sub-agents

Especialización

Los sub-agents son tipos de agente predefinidos con herramientas y comportamientos especializados. Se crean con subagent_type en el tool Agent. El sistema incluye tipos como Explore, Plan, claude-code-guide, entre otros.

Sub-agent Explore — búsqueda rápida 
// Explore: solo tiene herramientas de lectura (no escribe código)
// Ideal para localizar archivos o entender la estructura

Agent({
  subagent_type: "Explore",
  description: "Encontrar dónde se define la autenticación JWT",
  prompt: `Busca en el codebase:
  1. Dónde se genera el JWT (sign)
  2. Dónde se valida (verify)
  3. Qué middleware lo aplica
  Búsqueda: "very thorough". Devuelve rutas + líneas.`
});
Sub-agent Plan — diseño de arquitectura 
// Plan: analiza el problema y devuelve un plan paso a paso
// No escribe código, solo planifica

Agent({
  subagent_type: "Plan",
  description: "Plan para migrar a PostgreSQL",
  prompt: `Analiza el codebase actual (usa SQLite con better-sqlite3).
  Diseña un plan detallado para migrar a PostgreSQL:
  - Qué archivos cambiar y en qué orden
  - Cómo manejar las migraciones de esquema
  - Qué tests añadir para validar la migración
  - Riesgos y cómo mitigarlos
  Devuelve pasos numerados con archivos específicos.`
});
Definir un sub-agent propio .claude/agents/code-reviewer.md 
---
name: code-reviewer
description: >
  Revisa código buscando bugs de correctitud y oportunidades de
  simplificación. Devuelve hallazgos priorizados por severidad.
tools:
  - Read
  - Bash
  - WebSearch
model: claude-opus-4-8
---

Eres un code reviewer senior. Tu objetivo es encontrar:
1. Bugs reales (no hipotéticos)
2. Código que puede simplificarse sin perder claridad
3. Patrones que violan principios SOLID

Para cada hallazgo: archivo:línea · severidad (high/med/low) · descripción · fix sugerido.
Sé conciso. Omite hallazgos de baja confianza.
Usar el sub-agent custom 
Agent({
  subagent_type: "code-reviewer",
  description: "Review del módulo de auth",
  prompt: `Revisa src/auth/*.ts.
  Enfócate en: manejo de errores,
  validación de inputs, y uso correcto
  de async/await.`
});
Resultado del sub-agent 
**HIGH** src/auth/jwt.ts:42
Unhandled promise rejection en verify().
Fix: wrap en try/catch o usa .catch()

**MED** src/auth/middleware.ts:18
req.user asignado sin type guard.
Fix: validar con zod antes de asignar

**LOW** src/auth/session.ts:91
Lógica duplicada con token.ts:34.
Fix: extraer a validateToken()
08

Harness

Infraestructura

El harness es el entorno de ejecución administrado que corre Claude Code on the Web. Es el contenedor efímero, el sistema de hooks, los permisos, las variables de entorno y la sesión completa. Tú configuras el harness; él ejecuta los hooks, gestiona el ciclo de vida y da a Claude acceso al repo.

Trigger (PR / web / API) → Harness spawns container → SessionStart hooks → Claude actúa → Per-tool hooks → Stop hook + push
Configuración completa del harness .claude/settings.json 
{
  "model": "claude-opus-4-8",

  "env": {
    "NODE_ENV": "test",
    "LOG_LEVEL": "debug"
  },

  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git *)",
      "Read(**)",
      "Edit(**)",
      "Write(**)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)"
    ]
  },

  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "npm ci --silent && echo '✓ deps instaladas'"
      }]
    }],

    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npx tsc --noEmit 2>&1 | tail -5"
      }]
    }],

    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "npm test -- --passWithNoTests 2>&1 | tail -10"
      }]
    }]
  }
}
CLAUDE.md — memoria persistente del proyecto CLAUDE.md 
# Proyecto API

## Stack
- Node.js 20 + TypeScript 5.4
- PostgreSQL (via prisma)
- Vitest para tests

## Comandos frecuentes
- `npm run dev`    → servidor local en :3000
- `npm test`       → tests unitarios
- `npm run db:migrate` → migraciones pendientes

## Reglas importantes
- Nunca commitear a main directamente
- Todos los endpoints deben tener tests de integración
- Usar Zod para validar inputs externos

## Contexto de arquitectura
El servicio de auth vive en src/auth/ y usa JWT RS256.
Las claves están en AWS Secrets Manager, no en .env.
09

Todo junto

Sistema completo

Un sistema real combina todos los conceptos. Aquí: un flujo de CI automático donde un trigger de GitHub lanza un harness que usa hooks, MCP tools, sub-agents paralelos, y un skill final para generar el reporte.

GitHub PR abierto → Harness + SessionStart hook → Skill /review lanzada → 3 sub-agents paralelos → MCP GitHub comenta PR → Stop hook notifica Slack
1. Harness + hooks .claude/settings.json 
{
  "model": "claude-opus-4-8",

  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}" }
    }
  },

  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "npm ci --silent && npx tsc --noEmit && echo 'entorno listo'"
      }]
    }],
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "curl -s -X POST $SLACK_WEBHOOK -d '{\"text\":\"Review de PR completado\"}'"
      }]
    }]
  }
}
2. Skill /review orquesta todo .claude/skills/review.md 
---
description: "Review completo de un PR: código, tests y seguridad"
---

Primero obtén el diff del PR actual con `git diff main...HEAD`.

Lanza estos 3 sub-agents EN PARALELO (mismo mensaje):

**Agent 1 — Code quality**:
  subagent_type: Explore
  "Analiza el diff para calidad de código: complejidad, duplicación,
   nombres confusos. Lista hallazgos por archivo:línea."

**Agent 2 — Test coverage**:
  subagent_type: Explore
  "Revisa si los cambios tienen tests adecuados. Para cada función
   nueva o modificada, verifica si existe un test. Lista gaps."

**Agent 3 — Security**:
  subagent_type: code-reviewer
  "Revisa el diff buscando vulnerabilidades OWASP Top 10,
   secrets expuestos, inputs sin validar."

Cuando los 3 terminen, consolida sus resultados en un único
comentario de PR usando el tool mcp__github__add_issue_comment:
- Sección por categoría
- Emoji por severidad (🔴 high, 🟡 med, 🟢 low)
- Tabla resumen al final
3. Sub-agent custom: code-reviewer .claude/agents/code-reviewer.md 
---
name: code-reviewer
description: "Especialista en seguridad de código"
tools: [Read, Bash, WebSearch]
model: claude-opus-4-8
---

Eres un experto en seguridad de aplicaciones web.
Analiza código buscando: inyecciones, auth bypass, IDOR,
secrets hardcodeados, dependencias con CVEs.
Responde solo con hallazgos reales y concretos, priorizados.
4. Resultado: comentario en el PR 
## Review automático — PR #142

### 🔴 Seguridad (1)
| Archivo | Línea | Hallazgo |
|---|---|---|
| src/api/users.ts | 88 | SQL construido con string concat — usa parámetros |

### 🟡 Calidad (2)
| Archivo | Línea | Hallazgo |
|---|---|---|
| src/auth/session.ts | 34 | Función >50 líneas, considerar split |
| src/utils/format.ts | 12 | Duplica lógica de src/utils/date.ts:8 |

### 🟢 Tests (1)
| Archivo | Función | Estado |
|---|---|---|
| src/api/users.ts | updateUser() | Sin test de caso error 404 |

**Resumen:** 1 high · 2 medium · 1 low — requiere fix antes de merge
🏗️
Arquitectura del sistema: El harness gestiona el ciclo de vida. Los hooks automatizan el entorno y las notificaciones. La skill orquesta el flujo. Los sub-agents trabajan en paralelo con contexto limpio. El MCP publica los resultados. Cada pieza tiene una sola responsabilidad.
