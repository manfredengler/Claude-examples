Claude Concepts
De menos a más

Cada concepto del ecosistema Claude Code explicado con ejemplos reales y ejecutables. Empezamos con las primitivas básicas y terminamos combinándolas todas.

01

Tools

Primitiva

Los tools son las acciones concretas que Claude puede ejecutar: leer un archivo, correr bash, editar código, buscar en la web. Sin tools, Claude solo puede devolver texto. Con tools, puede actuar sobre el sistema.

💡
Analogía: Si Claude es un programador, los tools son su teclado, terminal y editor. Los built-in tools (Read, Bash, Edit, Write) siempre están disponibles. Tools adicionales vienen por MCP o por el Agent SDK.
Claude llama al tool 
{
  "tool": "Bash",
  "input": {
    "command": "git log --oneline -5",
    "description": "Last 5 commits"
  }
}
El sistema responde 
a3f1c2e fix: null pointer in auth
9b2e451 feat: add retry logic
c77f830 docs: update README
4d1a9f2 refactor: split services
8eae001 init: project scaffold
Ejemplo real — tool Edit src/config.ts 
// Claude recibe: "cambia el timeout a 30 segundos"
// Claude llama a Edit con old_string / new_string:

{
  "tool": "Edit",
  "input": {
    "file_path": "/src/config.ts",
    "old_string": "const TIMEOUT_MS = 5000;",
    "new_string": "const TIMEOUT_MS = 30_000;"
  }
}
Tools built-in más usados 
Read      → leer archivos (con offset/limit para archivos grandes)
Write     → crear archivos nuevos o reescritura completa
Edit      → reemplazar string exacto dentro de un archivo
Bash      → ejecutar comandos shell (git, npm, curl…)
Agent     → lanzar un sub-agente con su propio contexto
WebSearch → buscar en internet (requiere permiso)
WebFetch  → leer una URL concreta
02

Commands

Built-in + Custom

Los commands son shortcuts prefijados con / que se pueden escribir en el chat. Los built-in vienen con Claude Code. Los custom se crean como archivos .md en .claude/commands/.

Comandos built-in principales 
/help          # lista de comandos disponibles
/clear         # limpia el contexto de la conversación
/compact       # compacta el historial para ahorrar tokens
/memory        # edita el archivo CLAUDE.md del proyecto
/config        # abre la configuración interactiva
/doctor        # diagnóstico del entorno Claude Code
/cost          # muestra el costo de tokens de la sesión
/review        # code review del diff actual (skill)
/init          # genera CLAUDE.md inicial para el repo
Crear un comando custom .claude/commands/fix-ts.md 
---
description: "Corrige todos los errores de TypeScript en el proyecto"
---

Corre `npx tsc --noEmit` y captura la salida.
Por cada error encontrado:
1. Lee el archivo afectado con el tool Read
2. Entiende el contexto del error
3. Aplica la corrección mínima con Edit
4. Vuelve a correr tsc para verificar que se resolvió

Al final lista cuántos errores había y cuántos se corrigieron.
⚡
Una vez creado el archivo, el comando /fix-ts queda disponible en cualquier sesión dentro de ese proyecto. Puede recibir argumentos: /fix-ts src/auth.ts.
Comando con argumento $ARGUMENTS .claude/commands/explain.md 
---
description: "Explica una función o archivo en detalle"
---

Lee el archivo o función indicada: $ARGUMENTS

Explica:
- Qué hace (propósito de alto nivel)
- Cómo funciona (flujo paso a paso)
- Por qué está escrito así (decisiones de diseño no obvias)
- Posibles puntos de mejora

Sé conciso. Usa ejemplos cuando ayuden.
03

Hooks

Automatización

Los hooks son comandos shell que el harness ejecuta automáticamente en respuesta a eventos de Claude: antes de usar un tool, después de usarlo, al iniciar sesión, etc. Claude no los controla — los ejecuta el sistema.

Eventos disponibles 
PreToolUse    → antes de que Claude ejecute cualquier tool
PostToolUse   → después de que un tool termina
SessionStart  → al inicio de cada sesión
Stop          → cuando Claude termina su turno
Hook: lint automático tras cada Edit .claude/settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix \"${TOOL_INPUT_FILE_PATH}\" 2>&1 | head -20"
          }
        ]
      }
    ]
  }
}
Hook: log de todos los comandos Bash .claude/settings.json 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[$(date -u +%H:%M:%S)] BASH: $TOOL_INPUT_COMMAND\" >> .claude/bash-audit.log"
          }
        ]
      }
    ]
  }
}
Hook SessionStart: setup del entorno .claude/settings.json 
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm install --silent && echo 'deps ready'"
          }
        ]
      }
    ]
  }
}
🔒
Seguridad: los hooks pueden bloquear un tool devolviendo exit code distinto de 0. Útil para prevenir rm -rf o push a main sin confirmación.
04

Skills

Web / Avanzado

Las skills son comandos slash con inteligencia integrada. A diferencia de los commands simples (texto plano), las skills tienen frontmatter que las registra en el sistema, pueden incluir lógica condicional y se usan especialmente en Claude Code on the Web.

Skill: security review .claude/skills/security-review.md 
---
description: >
  Revisa el diff actual buscando vulnerabilidades de seguridad
  (OWASP Top 10, secrets expuestos, inyecciones, etc.)
trigger: manual
---

## Pasos

1. Corre `git diff HEAD` para ver los cambios actuales
2. Para cada archivo modificado:
   - Lee el archivo completo con Read
   - Analiza el diff en contexto
3. Busca específicamente:
   - SQL/command injection
   - Secrets o credenciales hardcodeadas
   - XSS en templates HTML
   - Dependencias con CVEs conocidos (`npm audit`)
   - Permisos excesivos en archivos
4. Por cada hallazgo: indica archivo, línea, severidad y fix sugerido
5. Termina con un resumen: PASS / WARN / FAIL
Skill: deploy con validaciones .claude/skills/deploy.md 
---
description: "Deploy a producción con checklist automático"
trigger: manual
---

Antes de deployar, valida que:
- [ ] Tests pasan: `npm test`
- [ ] Build compila: `npm run build`
- [ ] No hay secrets en el código: `git log -p | grep -i "api_key\|password\|secret"`
- [ ] La rama es `main` o `release/*`

Si todo pasa, ejecuta el deploy: `npm run deploy`
Si algún check falla, detente y reporta qué falló.
🎯
Skills vs Commands: Los commands son instrucciones. Las skills son procedimientos con contexto, validaciones y flujo de decisión. Se invocan igual (/deploy), pero tienen más poder expresivo.
05

MCP Servers

Extensibilidad

MCP (Model Context Protocol) permite conectar Claude a sistemas externos como GitHub, bases de datos, Slack, APIs internas. Cada servidor MCP expone tools adicionales que Claude puede llamar igual que los built-in.

Configurar MCP en el proyecto .claude/settings.json 
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "${DATABASE_URL}"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
    }
  }
}
Claude llama tool MCP GitHub 
{
  "tool": "mcp__github__create_pull_request",
  "input": {
    "owner": "acme",
    "repo": "api",
    "title": "fix: auth timeout",
    "head": "feature/fix-auth",
    "base": "main",
    "body": "Increases timeout to 30s"
  }
}
Respuesta del servidor MCP 
{
  "number": 142,
  "url": "https://github.com/acme/api/pull/142",
  "state": "open",
  "created_at": "2026-06-02T10:30:00Z"
}
Crear un MCP server propio (Node.js) mcp-server/index.ts 
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "mi-empresa", version: "1.0.0" });

// Tool personalizado: consultar métricas internas
server.tool(
  "get_metrics",
  "Obtiene métricas de rendimiento del sistema",
  {
    service: z.string().describe("nombre del servicio"),
    range: z.enum(["1h", "24h", "7d"]).default("1h"),
  },
  async ({ service, range }) => {
    const data = await fetchInternalMetrics(service, range);
    return {
      content: [{ type: "text", text: JSON.stringify(data, null, 2) }],
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
06

Agents

Paralelismo

Un agent es una instancia de Claude lanzada por el agente principal mediante el tool Agent. Corre con su propio contexto, tools y modelo. El padre lo lanza y recibe el resultado al terminar: un string con el último mensaje del agente, o un objeto { path, branch } si se usó isolation: "worktree".

🧠
Regla de oro: usa agents cuando la tarea consumiría demasiado contexto, cuando el trabajo es paralelizable, o cuando quieres aislar una búsqueda para que no contamine el hilo principal.
Parámetros del tool Agent 
Agent({
  // REQUERIDO — qué hace este agente (aparece en el log del padre)
  description: "Migrar tests de Jest a Vitest",

  // REQUERIDO — el mensaje completo que recibe el agente.
  // Debe ser autosuficiente: el agente NO ve el contexto del padre.
  prompt: `...`,

  // OPCIONAL — tipo de agente especializado (ver sección 07)
  subagent_type: "Explore",

  // OPCIONAL — modelo específico para esta tarea
  // claude-opus-4-8 | claude-sonnet-4-6 | claude-haiku-4-5-20251001
  model: "claude-opus-4-8",

  // OPCIONAL — crea un git worktree aislado para el agente.
  // Si hace cambios → devuelve { path, branch }
  // Si no hace cambios → worktree se limpia automáticamente
  isolation: "worktree",

  // OPCIONAL — no bloquea al padre; notifica al terminar
  run_in_background: true,
});
Ciclo de vida de un agent 
Padre llama Agent({prompt: "..."})
  │
  ├─ El sistema crea una sesión nueva con contexto vacío
  ├─ El agente recibe solo el `prompt` como mensaje inicial
  ├─ Tiene acceso a los tools configurados (o los del tipo)
  ├─ Trabaja de forma autónoma hasta terminar su turno
  │
  └─ Devuelve al padre:
       • string  → su último mensaje de texto (caso normal)
       • { path, branch } → si se usó isolation: "worktree"
                            y el agente hizo cambios en disco

IMPORTANTE:
  - El agente NO ve el historial de conversación del padre
  - El padre NO ve los tool calls intermedios del agente
  - Solo se comparte el resultado final
3 agentes en paralelo → resultado consolidado 
// Clave: los 3 Agent calls van en el MISMO mensaje del padre
// → corren simultáneamente, no secuencialmente

const [authResult, paymentsResult, usersResult] = await Promise.all([
  Agent({
    description: "Auditar módulo auth",
    prompt: `Eres un auditor de seguridad. Lee src/auth/ completo.
    Lista todas las funciones exportadas, sus dependencias externas,
    y cualquier riesgo de seguridad evidente. Máx 150 palabras.`,
  }),
  Agent({
    description: "Auditar módulo payments",
    prompt: `Eres un auditor de seguridad. Lee src/payments/ completo.
    Lista todas las funciones exportadas, sus dependencias externas,
    y cualquier riesgo de seguridad evidente. Máx 150 palabras.`,
  }),
  Agent({
    description: "Auditar módulo users",
    prompt: `Eres un auditor de seguridad. Lee src/users/ completo.
    Lista todas las funciones exportadas, sus dependencias externas,
    y cualquier riesgo de seguridad evidente. Máx 150 palabras.`,
  }),
]);

// El padre consolida los 3 resultados en un reporte único
Worktree aislado — el agente trabaja en su propia rama 
const result = await Agent({
  description: "Migrar tests a Vitest",
  isolation: "worktree",
  prompt: `Contexto: el proyecto usa Jest. Tu tarea es migrarlo a Vitest.

  Pasos:
  1. Lee package.json para entender la config de Jest actual
  2. Para cada archivo *.test.ts en src/:
     a. Lee el archivo
     b. Reemplaza imports: jest → vitest (describe, it, expect, vi)
     c. Actualiza mocks: jest.fn() → vi.fn(), jest.mock() → vi.mock()
     d. Usa Edit para aplicar los cambios
  3. Actualiza package.json: reemplaza jest por vitest en scripts y deps
  4. Corre: npx vitest run 2>&1 | tail -20
  5. Si hay fallos, corrígelos antes de terminar
  6. Haz commit: "migrate: Jest → Vitest"

  Trabaja exhaustivamente en TODOS los archivos. No te saltes ninguno.`,
});

// Si el agente hizo cambios:
// result = { path: "/tmp/worktree-abc", branch: "agent/migrate-vitest" }
// Si no hizo nada:
// el worktree se limpia automáticamente
Background agent — no bloquea al padre 
// Lanzar sin esperar el resultado
Agent({
  description: "Generar JSDoc completo",
  run_in_background: true,
  prompt: `Añade JSDoc a todas las funciones públicas en src/.
  Para cada archivo: Read → analiza → Edit para añadir comentarios.
  Una línea por función: qué hace, parámetros, retorno.
  No documentes funciones privadas ni triviales (getters simples).`,
});

// El padre continúa trabajando...
// Cuando el agente termina, llega una notificación automática
// con su resultado — el padre puede actuar sobre él en ese momento.
✍️
Cómo escribir el prompt de un agent: el agente empieza sin contexto. El prompt debe incluir: qué archivos leer, qué herramientas usar, qué output se espera, y restricciones importantes. Un prompt vago produce resultados vagos. Incluye rutas concretas, no "los archivos relevantes".
07

Definir Agentes Custom

Especialización

Los agentes custom son archivos .md que definen un tipo de agente reutilizable: su system prompt, los tools que puede usar, y el modelo. Se guardan en .claude/agents/ (proyecto) o ~/.claude/agents/ (global). Una vez definidos se invocan con subagent_type: "nombre".

Anatomía completa de un agente custom .claude/agents/mi-agente.md 
---
# REQUERIDO: identificador que se usa en subagent_type
name: mi-agente

# REQUERIDO: descripción visible en el sistema (1-2 líneas)
# El sistema usa esto para decidir cuándo sugerir este agente
description: >
  Especialista en X. Úsalo cuando necesites Y.
  Devuelve Z en formato W.

# OPCIONAL: lista de tools permitidos (omitir = todos los tools)
# Built-in disponibles: Bash, Read, Write, Edit, Agent,
#   WebSearch, WebFetch
tools:
  - Read
  - Bash
  - WebSearch

# OPCIONAL: modelo para este agente
# claude-opus-4-8   → más capaz, más lento
# claude-sonnet-4-6 → balance (default)
# claude-haiku-4-5-20251001 → más rápido, tareas simples
model: claude-opus-4-8
---

Todo lo que está aquí es el SYSTEM PROMPT del agente.
Define su personalidad, metodología, y formato de respuesta.

El agente arranca leyendo este system prompt antes de recibir
el `prompt` del padre. No ve ningún otro contexto.
Agente de proyecto .claude/agents/ 
Disponible solo en este repositorio.
Ideal para agentes específicos del dominio:
revisores de reglas de negocio,
generadores de código del stack
concreto del proyecto, etc.

Se commitea al repo → todo el equipo
lo tiene disponible.
Agente global ~/.claude/agents/ 
Disponible en todos tus proyectos.
Ideal para agentes de propósito general:
tu code reviewer preferido, tu
generador de tests, tu escritor
de documentación.

No se comparte con el equipo — es
personal.
Agentes built-in del sistema 
Explore
  Tools: Bash(read-only), Read, WebFetch, WebSearch
  Uso: localizar código, encontrar archivos, responder
       preguntas sobre la estructura. NO escribe.
  Ideal para: búsquedas amplias sin riesgo de modificar nada.

Plan
  Tools: todos excepto Edit/Write/Bash(mutating)
  Uso: diseñar estrategias de implementación, identificar
       archivos críticos, analizar trade-offs.
  Ideal para: "¿cómo debería implementar X?" antes de hacerlo.

general-purpose
  Tools: todos (*)
  Uso: catch-all para tareas que no encajan en otro tipo.
  Es el default cuando no se especifica subagent_type.

claude-code-guide
  Tools: Bash, Read, WebFetch, WebSearch
  Uso: preguntas sobre Claude Code CLI, API, SDK, hooks,
       MCP servers, comandos, configuración.
Ejemplo 1 — Agente revisor de código .claude/agents/code-reviewer.md 
---
name: code-reviewer
description: >
  Revisa código buscando bugs reales y oportunidades de simplificación.
  Devuelve hallazgos priorizados. No escribe código.
tools:
  - Read
  - Bash
model: claude-opus-4-8
---

Eres un senior engineer revisando código de producción.

SOLO reportas hallazgos con alta confianza:
- Bugs concretos (null pointer, race condition, error swallowed)
- Código que viola una invariante del sistema
- Simplificaciones evidentes (duplicación clara, abstracción prematura)

Para cada hallazgo: `archivo:línea | severidad | descripción | fix`
Severidad: HIGH (bug que causa error) · MED (deuda técnica) · LOW (nitpick)

Omite: sugerencias de estilo, preferencias personales, "considera usar X".
Sé directo. Si no hay nada real que reportar, di "LGTM".
Ejemplo 2 — Agente generador de tests .claude/agents/test-writer.md 
---
name: test-writer
description: >
  Genera tests unitarios y de integración para funciones dadas.
  Usa el framework de tests del proyecto (detecta automáticamente).
tools:
  - Read
  - Write
  - Edit
  - Bash
model: claude-sonnet-4-6
---

Antes de escribir cualquier test:
1. Lee el archivo que contiene la función a testear
2. Lee un test existente del proyecto para entender el estilo y framework
3. Detecta el runner: Jest, Vitest, Mocha (busca en package.json)

Al escribir tests:
- Golden path primero, luego edge cases
- Un `describe` por función, un `it` por comportamiento
- Nombres en español: "debería retornar X cuando Y"
- Sin mocks salvo que sean estrictamente necesarios
- Siempre corre los tests al final: si fallan, corrígelos

Output: el archivo de tests creado y el resultado de correrlos.
Ejemplo 3 — Agente de investigación web ~/.claude/agents/researcher.md 
---
name: researcher
description: >
  Investiga un tema técnico con múltiples fuentes, verifica claims
  contradictorios y entrega un reporte citado. Úsalo para preguntas
  que requieren evidencia externa, no solo conocimiento del modelo.
tools:
  - WebSearch
  - WebFetch
model: claude-opus-4-8
---

Eres un investigador técnico riguroso.

Metodología:
1. Planifica 3-5 queries de búsqueda distintas para el tema
2. Ejecuta las búsquedas, anota las fuentes más relevantes
3. Fetchea las páginas con más contenido sustantivo
4. Identifica afirmaciones contradictorias entre fuentes
5. Si hay contradicción, busca una tercera fuente que desempate

Formato del reporte:
- Resumen ejecutivo (3 líneas)
- Hallazgos principales (bullets con citas)
- Puntos de controversia o incertidumbre
- Fuentes consultadas (URLs)
Invocar el agente custom 
// El padre invoca por nombre exacto del frontmatter
Agent({
  subagent_type: "code-reviewer",
  description: "Review del módulo de auth",
  prompt: `Revisa src/auth/*.ts en profundidad.
  El contexto: este módulo maneja autenticación JWT
  con refresh tokens. Enfócate especialmente en:
  - Manejo de tokens expirados
  - Validación de firma
  - Gestión de sesiones concurrentes`,
});
Output del agente 
src/auth/jwt.ts:42 | HIGH
verify() sin try/catch — un JWT malformado
lanza excepción no controlada al caller.
Fix: envolver en try/catch, retornar null.

src/auth/session.ts:88 | MED
refreshToken no invalida el token anterior
al emitir uno nuevo — posible replay attack.
Fix: añadir jti al blacklist antes de sign().

src/auth/middleware.ts:31 | LOW
req.user tipado como `any`. Fix: usar
interfaz AuthUser con campos mínimos.
🔑
Restricción de tools: si defines tools: [Read, Bash], el agente solo puede usar esos dos. Esto es una garantía de seguridad: un agente de investigación sin Edit nunca puede modificar código por accidente, aunque el prompt se lo pida.
08

Harness

Infraestructura

El harness es el entorno de ejecución administrado que corre Claude Code on the Web. Es el contenedor efímero, el sistema de hooks, los permisos, las variables de entorno y la sesión completa. Tú configuras el harness; él ejecuta los hooks, gestiona el ciclo de vida y da a Claude acceso al repo.

Trigger (PR / web / API) → Harness spawns container → SessionStart hooks → Claude actúa → Per-tool hooks → Stop hook + push
Configuración completa del harness .claude/settings.json 
{
  "model": "claude-opus-4-8",

  "env": {
    "NODE_ENV": "test",
    "LOG_LEVEL": "debug"
  },

  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git *)",
      "Read(**)",
      "Edit(**)",
      "Write(**)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)"
    ]
  },

  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "npm ci --silent && echo '✓ deps instaladas'"
      }]
    }],

    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npx tsc --noEmit 2>&1 | tail -5"
      }]
    }],

    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "npm test -- --passWithNoTests 2>&1 | tail -10"
      }]
    }]
  }
}
CLAUDE.md — memoria persistente del proyecto CLAUDE.md 
# Proyecto API

## Stack
- Node.js 20 + TypeScript 5.4
- PostgreSQL (via prisma)
- Vitest para tests

## Comandos frecuentes
- `npm run dev`    → servidor local en :3000
- `npm test`       → tests unitarios
- `npm run db:migrate` → migraciones pendientes

## Reglas importantes
- Nunca commitear a main directamente
- Todos los endpoints deben tener tests de integración
- Usar Zod para validar inputs externos

## Contexto de arquitectura
El servicio de auth vive en src/auth/ y usa JWT RS256.
Las claves están en AWS Secrets Manager, no en .env.

Todo junto

Sistema completo

Un sistema real combina todos los conceptos. Aquí: un flujo de CI automático donde un trigger de GitHub lanza un harness que usa hooks, MCP tools, sub-agents paralelos, y un skill final para generar el reporte.

GitHub PR abierto → Harness + SessionStart hook → Skill /review lanzada → 3 sub-agents paralelos → MCP GitHub comenta PR → Stop hook notifica Slack
1. Harness + hooks .claude/settings.json 
{
  "model": "claude-opus-4-8",

  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}" }
    }
  },

  "hooks": {
    "SessionStart": [{

  subagent_type: Explore
  "Revisa si los cambios tienen tests adecuados. Para cada función

  subagent_type: code-reviewer
  "Revisa el diff buscando vulnerabilidades OWASP Top 10,
